# AMI qrels bias audit

## Summary

`data/eval/ami_qrels.json` has been expanded from 10 to 20 queries. Each query still has 3 relevant meeting IDs.

The original 10 queries are kept for continuity, but they are not a reliable standalone benchmark: they were created during the Sprint 1 MVP and are strongly biased toward the baseline semantic embedding search.

The added 10 queries were selected from corpus/chunk content inspection rather than from embedding top results. This makes the test less circular, but it is still not a blind human-labeled benchmark.

## Validity checks

Audit output: `evaluation/results/qrels_bias_audit_ami.json`.

| Check | Result |
|-------|--------|
| Query count | 20 |
| Relevant IDs per query | 3 |
| Missing relevant meeting IDs | 0 |
| Avg relevant-term coverage | 0.9012 |
| Original 10 term coverage | 0.9500 |
| Added 10 term coverage | 0.8524 |

High term coverage means many query terms appear directly in relevant meetings. This is useful for initial validation, but it also means the qrels still have lexical/topic-selection bias and are not a pure semantic generalization test.

## Bias signal: original vs added queries

| Method | Bucket | Precision@5 | Recall@5 | MRR@5 |
|--------|--------|-------------|----------|-------|
| embedding | original 10 | 0.6000 | 1.0000 | 1.0000 |
| embedding | added 10 | 0.0800 | 0.1333 | 0.1750 |
| multi_query_rrf | original 10 | 0.4600 | 0.7667 | 0.9333 |
| multi_query_rrf | added 10 | 0.0600 | 0.1000 | 0.1500 |
| llm_multi_query_rrf | original 10 | 0.4600 | 0.7667 | 1.0000 |
| llm_multi_query_rrf | added 10 | 0.0400 | 0.0667 | 0.1250 |

This split confirms the original 10 are too easy or biased toward the current retrieval setup. The added 10 are much harder for all methods.

However, the gap is so large that it also warns us the added labels may be incomplete or noisy. Some added queries describe broad concepts such as visual appearance, marketing positioning, and detailed design. Many AMI meetings may be partially relevant, so limiting each query to exactly 3 relevant IDs may under-label true positives.

## Is the 20-query set general?

Better than before, but not fully general.

Improvements:

- More topics: materials, casing, visual style, target users, child lock/timer, manufacturing, marketing, finance.
- Less circular: added labels were not copied from embedding top results.
- Metrics are no longer saturated at 1.0.

Remaining bias/risk:

- Still only 20 queries.
- Still exactly 3 relevant IDs per query, which can under-label broad topics.
- Added labels came from corpus inspection and keyword/topic search, not blind annotation.
- Query wording still often shares terms with relevant transcripts.
- No negative/ambiguous queries yet.
- No chunk-level relevance labels yet.

## Recommendation

Use the 20-query file as a stronger smoke test, not as final benchmark evidence.

For a more trustworthy benchmark, create 50-100 qrels with:

- blind human labeling from candidate pools generated by multiple systems, not only embedding;
- at least 5-10 paraphrase-heavy queries;
- ambiguous queries and negative/no-answer queries;
- variable number of relevant meetings per query;
- chunk-level relevance for snippet quality;
- nDCG@K and Recall@1/3/5 in addition to Precision/Recall/MRR@5.
